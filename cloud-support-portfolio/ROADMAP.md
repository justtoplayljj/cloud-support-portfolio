# ROADMAP.md

本路线图用于规划 Cloud Support Portfolio 的建设顺序。具体执行项以 `TASKS.md` 为准；路线图关注阶段目标、交付成果和验收条件，不绑定固定日期。

## 当前状态

仓库已具备基础目录结构、架构图、10 篇 Incident、9 篇 Runbook 和 5 个 Shell 脚本。当前状态如下：

- 原有 Incident、Runbook 和自动化脚本空文件已全部补全。
- `automation/*.sh` 已通过 Bash 语法检查。
- `README.md` 已补充实际文件链接和新增内容索引。
- Incident 与 Runbook 的统一深度审查仍在进行中。
- 监控、Kubernetes 清单、工单和复盘目录尚未形成可展示内容。
- 尚未建立自动化测试、ShellCheck 和持续集成流程。

## 阶段 1：建立可用基线

状态：基本完成。

### 目标

消除阻塞问题，确保仓库中的现有内容准确、可读、可执行。

### 交付物

- 修复 `automation/backup.sh` 的语法错误。
- 补全 `docker-cleanup.sh`、`rotate-log.sh` 和 `system-report.sh`。
- 修正 README 中失效或错误的路径、文件名和 Incident 编号。
- 修正 `incidents/003-nginx-502.md` 的内部编号。
- 建立统一的文档结构和 Shell 编码规范。

### 验收标准

- `bash -n automation/*.sh` 全部通过。
- README 中引用的本地文件全部存在。
- 仓库中没有未说明用途的空脚本。
- 所有现有 Incident 编号与文件名一致。

## 阶段 2：完成核心故障排查内容

状态：内容已补全，统一审查待完成。

### 目标

形成覆盖 Linux、网络、Web、容器和 Kubernetes 的核心故障案例与操作手册。

### 交付物

- 完成高 CPU、DNS 故障、证书过期、容器崩溃和内存泄漏 Incident。
- 完成节点离线、Pod CrashLoop 和 Ingress 故障 Incident。
- 完成 CPU、磁盘、DNS、网络和 SSL 排查 Runbook。
- 完成 Kubernetes Pod 排查 Runbook。
- 统一 Incident 和 Runbook 的章节、术语及命令展示方式。

### 验收标准

- 每篇 Incident 包含现象、影响、调查、根因、恢复和预防措施。
- 每篇 Runbook 包含适用范围、前置条件、步骤、验证和回滚方案。
- 示例命令可执行，模拟内容有明确标识。
- README 提供可点击的完整内容索引。

## 阶段 3：提高自动化质量

### 目标

将现有脚本提升为安全、可重复执行、可验证的运维工具。

### 交付物

- 完善健康检查和备份脚本的参数校验、错误处理及退出码。
- 为清理、删除和覆盖操作增加 dry-run 或安全确认机制。
- 为关键脚本增加最小测试用例。
- 建立统一验证脚本 `scripts/validate.sh`。
- 引入 ShellCheck 和 Markdown 本地链接检查。

### 验收标准

- 所有 Shell 脚本通过 Bash 语法检查。
- ShellCheck 不存在未处理的高严重度问题。
- 关键脚本覆盖成功、错误参数和依赖缺失场景。
- 单条验证命令可以完成主要质量检查。

## 阶段 4：扩展云支持作品集

### 目标

补充监控、Kubernetes、技术支持和事故复盘内容，展示端到端问题处理能力。

### 交付物

- Prometheus 监控架构和指标说明。
- Grafana Dashboard 配置说明及脱敏截图。
- Kubernetes 部署清单和运维示例。
- 至少一篇完整 Postmortem。
- 至少两个脱敏的技术支持工单案例。
- 明确 `cloud/` 目录定位，并补充云平台相关实践内容。
- 更新 Draw.io 架构源文件和导出的 PNG。

### 验收标准

- 监控内容包含指标、告警、诊断路径和 Dashboard 说明。
- Kubernetes 示例可通过静态校验，并提供部署与清理步骤。
- Postmortem 包含时间线、影响、根因、修复和行动项。
- 工单案例展示问题澄清、证据收集、沟通和闭环过程。

## 阶段 5：建立持续集成与发布流程

### 目标

通过自动检查保持仓库长期可维护，并形成稳定的作品集发布版本。

### 交付物

- CI 工作流：Bash 语法检查、ShellCheck 和 Markdown 链接检查。
- 敏感信息扫描。
- 发布前检查清单。
- 项目版本标签和变更记录约定。
- README 最终整理，包括项目亮点、架构、案例索引和验证方式。

### 验收标准

- Pull Request 或提交可自动发现语法错误、失效链接和敏感信息。
- 主分支质量检查全部通过。
- 新环境可按照 README 完成内容浏览和示例验证。
- 仓库不存在未说明用途的空文件或空目录。

## 长期维护方向

- 定期增加来自真实场景抽象的 Incident、Runbook 和工单案例。
- 根据工具和平台版本变化更新命令、配置及截图。
- 为高频故障补充自动诊断或自动恢复脚本。
- 跟踪 `TASKS.md` 中的行动项，完成后及时更新状态。
- 所有新增内容继续遵循 `AGENTS.md` 的安全、文档和验证要求。

## 推荐执行顺序

1. 完成阶段 1，建立可靠基线。
2. 并行推进 Incident 与 Runbook，完成阶段 2。
3. 在脚本内容稳定后实施阶段 3。
4. 基于核心内容扩展阶段 4。
5. 最后以阶段 5 固化质量门禁和发布流程。
