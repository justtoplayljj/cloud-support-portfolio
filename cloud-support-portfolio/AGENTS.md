# AGENTS.md

## 项目定位

本仓库是面向 Cloud Support、Linux 运维、Technical Support 和初级 SRE 岗位的个人实践作品集。修改内容时应突出可复现的故障排查过程、明确的运维步骤和可验证的结果。

## 仓库结构

- `architecture/`：系统架构图及其可编辑源文件。
- `automation/`：Shell 自动化脚本。
- `docs/`：通用技术文档和故障排查指南。
- `incidents/`：故障案例、根因分析及恢复记录。
- `runbooks/`：可执行的标准操作流程（SOP）。
- `README.md`：项目入口和内容索引。

新增文件应放入最贴近其用途的目录，并同步更新 `README.md` 中的相关索引。架构图发生变化时，应同时更新源文件和导出的 PNG 文件。

## 内容规范

- 文档使用 Markdown，标题层级连续，步骤按执行顺序编写。
- 命令、文件路径、服务名和配置项使用反引号标记。
- Incident 文档应包含：现象、影响、排查过程、根因、恢复步骤和预防措施。
- Runbook 应包含：适用范围、前置条件、操作步骤、验证方法和回滚方案。
- 示例不得包含真实密码、令牌、私钥、内部地址或个人敏感信息；统一使用清晰的占位符。
- 不虚构执行结果、监控数据或故障证据。模拟数据必须明确标注。
- 保持现有文档的主要语言和术语风格；同一文件内避免中英文术语混用不一致。

## Shell 脚本规范

- 新脚本使用 Bash，并以 `#!/usr/bin/env bash` 开头。
- 默认启用 `set -euo pipefail`；若不适用，应在脚本中说明原因。
- 变量引用使用双引号，常量和可配置项集中定义，避免依赖隐式工作目录。
- 对服务重启、文件删除、备份覆盖等高风险操作提供参数校验、明确提示或安全的默认行为。
- 错误信息写入标准错误，失败时返回非零退出码。
- 脚本应支持重复执行，且不得把凭据或环境专属配置硬编码进仓库。

## 修改原则

- 只修改当前任务需要的文件，不顺带重写无关内容。
- 保留用户已有修改，不覆盖或回退来源不明的变更。
- 文件重命名或移动后，检查并更新仓库中的所有引用。
- 新增操作步骤时，同时提供成功判定方式；涉及风险时，同时说明回滚方式。
- README 中的文件名和链接必须与仓库实际路径一致。

## 验证要求

提交修改前，根据变更范围执行以下检查：

```bash
# 检查 Markdown 中引用的本地路径和文件名
rg -n 'architecture/|automation/|docs/|incidents/|runbooks/' README.md docs incidents runbooks

# 检查 Bash 语法
bash -n automation/*.sh

# 如已安装 ShellCheck，检查 Shell 脚本质量
shellcheck automation/*.sh
```

此外，应人工确认：

- Markdown 标题、列表和代码块渲染正常。
- 示例命令与文档所述操作一致。
- 新增或改名的内容已在 `README.md` 中正确链接。
- 未提交密钥、凭据、日志转储或其他敏感信息。

若某项检查因工具未安装或环境限制而无法执行，应在交付说明中明确指出。
